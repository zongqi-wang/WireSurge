WireSurge

A local-first desktop and CLI application for API exploration, protocol workflows, and high-rate traffic generation. The product combines a Yaak-like request workspace with a Flamethrower-like load engine, then extends both with multi-threaded execution, auto-ladder experiments, Git-native workflows, configurable metrics, and sandboxed plugins.

Summary

WireSurge is built as a Rust workspace that produces a standalone wiresurge executable for each supported operating system and CPU architecture. The CLI invokes the engine directly. The desktop app launches the same executable in an internal engine --ipc mode and exchanges asynchronous commands, events, metrics, and logs over local IPC. The process boundary gives the frontend a cooperative stop command and a reliable hard kill switch.

HTTP/API workflows are first-class alongside DNS workflows. Traffic is modeled as composable protocol stages, while runtime responsibilities are modeled as supervised actors and bounded task groups. Immutable input corpora are shared once, connections have one explicit owner, and every long-running operation participates in a cancellation tree.

Core Tenets

No account required

Every core feature works without sign-in, registration, license server checks, or hosted user identity.

No cloud dependency

Workflow editing, execution, reports, secrets, plugins, and history work locally. Cloud sync can be optional later.

No telemetry

No product analytics, tracking, crash uploads, or traffic metadata collection by default. Diagnostics are explicit exports.

Secure local secrets

Secrets are stored via the OS keychain. Sensitive values are obscured in the UI, logs, exported reports, and screenshots by default.

Explicit ownership

Every connection, queue, report writer, and run state machine has one owner. Cross-component communication uses bounded messages.

Graceful termination

UI cancellation, Ctrl-C, and SIGTERM enter the same shutdown state machine. Abrupt termination remains crash-recoverable.

Established libraries over handrolled code

Standards-heavy and security-sensitive work uses well-established first- or third-party libraries instead of bespoke reimplementations: TLS, HTTP/1.1 and HTTP/2, async runtime and signals, DNS and EDNS0 message encoding, PROXY protocol framing, JSON/YAML serialization, CLI parsing, and latency histograms. WireSurge writes its own code only for what it must own — scheduling, pacing, connection lifecycle, cancellation, and metrics — and for small, well-contained helpers. A handrolled parser, protocol codec, or arg scanner is a defect to remove, not a feature. See the Dependency Policy for the admission rules that enforce this.

Product Shape

The product has two user-facing surfaces and one internal engine mode. They share workflow files, schemas, protocol modules, lifecycle rules, and report formats.

Surface Purpose Must Have
Desktop app Explore APIs, compose workflows, run tests, inspect responses, visualize load results. Request editor, environment manager, live charts, logs, reports, Git-aware file browser.
CLI A human-first command-line tool: people run workflows, validate configs, and read load results at the terminal. Rich help, "did you mean" suggestions, and colored output are first-class; scripts and agents are also supported through an opt-in --output json machine mode. init, run, ladder, validate, report, secret, plugin; discoverable --help at every level; --output json for automation.

Product framing: WireSurge is not just an API client and not just a benchmark tool. It is a programmable traffic workbench. The UI helps humans design flows. The engine runs them at controlled or aggressive scale.

Language Choice

Recommendation: Rust core, Rust CLI, Tauri desktop shell, React UI. This gives the best balance of native performance, memory safety, concurrency, protocol control, and desktop distribution.

Portable means one self-contained release artifact per OS and CPU target, not one native binary that runs everywhere. Publish macOS universal, Linux x86_64/arm64, and Windows x86_64 artifacts. Keep the CLI independent from Tauri and avoid OpenSSL or other mandatory system runtime dependencies; use a Rust TLS stack.

Choice Fit Strengths Tradeoffs
Rust + Tauri + React Best overall Fast native engine, safe concurrency, good CLI, good protocol ecosystem, small desktop shell, shared app core. Steeper learning curve. UI build still needs TypeScript/React.
Go + Wails Good pragmatic fallback Fast delivery, simple concurrency, excellent static binaries. Less precise control for protocol parsing, TLS customization, and sandboxed extensibility.
C++ + Qt Powerful but costly Maximum low-level control and mature native UI. High complexity and more memory/concurrency risk.
Zig Interesting research option Low-level control and excellent cross-compilation story. Immature ecosystem for TLS, HTTP, DNS, UI, plugins, and metrics.
Electron/TypeScript engine Weak for this project Fast UI iteration. Wrong center of gravity for high-rate networking and protocol manipulation.

Dependency Policy

Use established libraries for standards-heavy and security-sensitive work, but import the narrowest stable layer. WireSurge owns scheduling, cancellation, connection lifecycle, pacing, and metrics. Libraries own protocol correctness, cryptography, serialization, storage primitives, and other concerns where a mature implementation reduces risk.

Candidate Decision Reasoning
NLnet Labs domain Selected for DNS messages Use the stable base message, name, record type, and EDNS APIs with only the std feature. Do not enable its experimental transport or resolver features.
Hickory hickory-proto Revisit for a concrete transport need Mature and capable, but its protocol-only dependency graph is materially broader. Do not carry both DNS stacks without an interoperability requirement and benchmark.

Direct crates are declared once in the workspace, pinned to reviewed versions, and built from a committed lockfile. Default features are disabled unless justified. CI rejects unknown registries and Git sources, wildcard versions, duplicate versions, unapproved licenses, yanked crates, and known RustSec advisories. Native code, build scripts, bundled executables, and new cryptographic implementations require an architecture decision record.

Workflow Model

Workflows should be file-native and Git-friendly. Use a strict schema in YAML or TOML, with generated JSON schema for validation and editor autocompletion. The UI should edit the same files the CLI runs. 

version: 1
name: api-and-dns-capacity

profiles:
  local:
    vars:
      api_base: "https://api.example.test"
      resolver: "1.1.1.1:853"

secrets:
  api_token:
    source: keychain
    key: example-api-token

variables:
  tenant:
    list: ["alpha", "beta", "gamma"]
  request_id:
    generator: uuid_v7

flows:
  fetch-user:
    kind: http
    stages:
      - http:
          method: GET
          url: "{{vars.api_base}}/v1/users/{{vars.tenant}}"
          headers:
            authorization: "Bearer {{secrets.api_token}}"
            x-request-id: "{{vars.request_id}}"
      - assert:
          status: 200
          latency_p95_ms_lt: 250

  dot-query:
    kind: dns
    stages:
      - tcp_connect:
          target: "{{vars.resolver}}"
      - tls:
          sni: "cloudflare-dns.com"
          alpn: ["dot"]
      - dns:
          qname: "{{vars.tenant}}.example.com"
          qtype: A
          edns0:
            udp_size: 1232
            options:
              - code: 65001
                payload_hex: "cafe"

experiments:
  find-maximum-load:
    type: auto_ladder
    flows: ["fetch-user", "dot-query"]
    climb:
      mode: qps_then_workers
      qps_start: 100
      qps_step: 100
      interval: 30s
      max_workers: 16
    stop_when:
      error_rate_gt: 0.01
      timeout_rate_gt: 0.005
      latency_p99_ms_gt: 750
    cool_down: 20s

Execution Engine

Tasks, threads, and ownership

Named responsibilities do not automatically receive dedicated operating-system threads. Nonblocking components run as supervised async tasks on a bounded Tokio runtime. Dedicated threads are reserved for blocking persistence and CPU-heavy generation when profiling shows they are needed.

Responsibility Execution model Owns
Supervisor Main async task Run state, child task lifecycle, IPC, root cancellation token.
Scheduler One async task per run Rate credits, ladder phase, work admission, stop conditions.
I/O runtime N work-stealing threads Polling sockets, timers, protocol futures, bounded task queues.
Generators Bounded CPU pool Domain index selection, payload mutation, packet encoding.
Connection managers Async actors TCP/TLS streams, HTTP pools, response correlation, idle expiry.
Metrics One aggregator task Counter deltas, histogram merging, runner snapshots.
Persistence Dedicated blocking task Atomic reports, structured logs, checkpoints, final flush.
  • All queues are bounded and expose saturation metrics.
  • Workflow compilation, variable expansion, secret resolution, and corpus indexing happen before the hot path.
  • Per-worker counters and histograms are merged periodically instead of sharing one hot lock.
  • Work items carry indexes and immutable references, not copied request or corpus objects.
  • Random seeds and scheduler decisions are recorded for reproducibility.

Backend tiers

Backend Target Use Notes
Portable async sockets macOS, Linux, Windows Default desktop and CLI execution. Best for correctness, developer experience, and cross-platform behavior.
Linux batch sockets Linux runners High-rate UDP/TCP workloads. Optional backend using batching, socket sharding, and per-core ownership while preserving the same engine contract.
Linux advanced I/O Linux runners Best-in-class packet rate experiments. Evaluate io_uring, AF_XDP, netmap, or DPDK behind feature flags after the workflow model is stable.

Shutdown and Signals

UI cancellation, Ctrl-C, Unix SIGINT/SIGTERM, and internal failures all enter one lifecycle state machine. The first request is cooperative; a repeated request or expired deadline becomes immediate. SIGKILL cannot be handled, so persistent state must be crash-recoverable without cleanup code. 

Running
  -> StopRequested        stop scheduling new work
  -> Draining             wait for in-flight work within grace period
  -> Flushing             merge metrics, close pools, write atomic report
  -> Stopped              join owned tasks and exit

Second signal / deadline:
  StopRequested | Draining | Flushing -> ForcedStop
  1. The supervisor cancels the root token; subsystem child tokens are cancelled automatically.
  2. The scheduler stops issuing work and closes producer sides of bounded queues.
  3. Connection actors reject new requests and drain or cancel in-flight protocol operations.
  4. Metrics publishes a final snapshot; persistence writes through a temporary file and atomic rename.
  5. The supervisor joins owned tasks in dependency order and returns a cancelled or failed run outcome.
  6. The desktop escalates from IPC cancellation to SIGTERM, then process kill if the sidecar misses both deadlines.

Never use std::process::exit as the normal shutdown path because it skips destructors. Forced termination is a fallback, not cleanup.

Corpus Memory and Randomization

Loading a million domains independently in every thread multiplies memory usage. WireSurge stores each corpus once and gives workers lightweight indexes into immutable shared data.

Memory-mapped domain file | +-- shared offset table: u32/u64 per row | +-- Arc<DomainCorpus> | +-------+-------+-------+ | | | worker 0 worker 1 worker N seed+counter seed+counter bounded buffers bounded buffers
  • Use a memory-mapped newline-delimited file or compact immutable blob with an offset table.
  • One million u32 offsets require roughly 4 MB; workers do not hold a million String objects.
  • Sampling with replacement maps a counter-based random value to a corpus index.
  • Visit-each-once mode uses a seeded permutation over indexes instead of allocating and shuffling a million-entry vector.
  • Workers consume disjoint counters or scheduler-assigned index ranges, then build only bounded in-flight payloads.
memory = corpus bytes
       + offset table
       + workers * bounded in-flight buffers

memory != workers * entire corpus

Connection Ownership and Reuse

A TCP or TLS stream is ordered, stateful, and protocol-coupled. It must have one owning connection actor. Workers submit requests through bounded channels or a pool API; they do not concurrently read and write the same stream behind a shared mutex.

Protocol Reuse model Concurrency rule
HTTP/1.1 Per-origin keep-alive pool. Lease one connection per in-flight request; avoid pipelining initially.
HTTP/2 Shared client pool with multiplexed streams. Many requests can share one TCP/TLS connection within peer stream limits.
Raw TCP/TLS One actor per connection or sharded pool. The owner serializes writes and maps parsed responses to callers.
DNS/UDP Socket per I/O worker or receiver/demultiplexer task. Correlate responses by transaction ID and destination metadata. 
connections:
  mode: pooled            # pooled | fresh | fixed
  max_per_origin: 32
  max_streams: 100        # HTTP/2
  idle_timeout: 30s
  tls_sessions: reuse     # reuse | fresh

Connection behavior is part of the experiment definition. A capacity test may reuse TLS sessions and pooled connections; a handshake test intentionally requests fresh connections. Metrics report both logical requests and physical connections.

Protocol Support

The workflow engine should model traffic as stages. Stages compose into flows. This gives HTTP/API a first-class experience while keeping a clean path toward custom network stacks.

Phase Protocol work Reason
V1 HTTP/1.1, HTTP/2, REST, JSON, GraphQL basics, DNS UDP/TCP/DoT, TLS SNI, EDNS0, PROXY protocol v1/v2. Matches the core "Yaak plus Flamethrower" promise with API and DNS as peers.
V1.5 DoH, WebSocket, SSE, gRPC unary, advanced HTTP auth, request imports from curl/OpenAPI/Postman. Improves API-client usefulness and test realism.
V2 Custom TLS extensions, packet templates, raw UDP payloads, pcap replay, Geneve/VXLAN research backend. Turns the app into a broader protocol lab without destabilizing V1.

Best-in-class packet rate and arbitrary network stack definition pull in different directions. Keep V1 focused on correct composable stages, then add specialized high-rate backends that implement selected stage combinations efficiently.

Plugin Model

Plugins should assume untrusted third parties. The owner still decides what to install, but the default runtime should limit blast radius. Start with WebAssembly plugins using WASI-style capabilities.

Plugin type Examples Default permissions
Generators Custom IDs, payload mutations, DNS labels, token refresh values. No filesystem, no network, deterministic random source unless granted.
Mutators HTTP signing, custom headers, EDNS0 payload construction, TLS parameter selection. Read only the stage context and declared variables.
Assertions Response checks, protocol checks, latency classification. Read response metadata and redacted body snippets.
Encoders Binary protocol frames, future tunnel metadata, custom payload codecs. No ambient host access. Explicit byte buffers in, byte buffers out.

Plugin controls

  • Manifest declares capabilities: network, filesystem, keychain, environment, subprocess, UI contribution.
  • Desktop install screen shows requested capabilities before enabling a plugin.
  • Plugins are disabled per workspace by default unless explicitly trusted.
  • Plugin output is marked tainted until validation passes.
  • Native plugins can be added later as out-of-process plugins with explicit trust prompts.

Metrics and Logs

Metrics must be cheap in the hot path and useful after the run. Prefer per-worker counters and histograms with periodic aggregation.

Live metrics

QPS, RPS, latency percentiles, errors, timeouts, open connections, bytes in/out, worker saturation.

Run reports

Summary, capacity curve, thresholds, system info, workflow hash, Git commit, redacted config snapshot.

Exports

JSON, NDJSON, CSV, HTML, Prometheus text format, OpenTelemetry metrics and traces where appropriate.

Logging

  • Configurable log levels per component: engine, protocol, plugin, workflow, metrics, UI.
  • Structured logs by default for CLI and runner.
  • Human-readable log panel in desktop app.
  • Redaction runs before persistence and display.
  • Opt-in debug capture can include sensitive values only with a clear warning and local-only storage.

Desktop and WASM

Build the UI as shared React components and package it inside Tauri. The desktop process launches the packaged wiresurge executable as a child in internal engine mode. The UI sends asynchronous commands over local IPC and subscribes to events; it never owns sockets, scheduler state, or mutable engine resources.

Desktop app

Workflow editor, keychain access, reports, logs, Git state, sidecar lifecycle, and an always-accessible Runners view.

WASM/web app

Workflow editor, docs examples, report viewer, sanitized demo runs, website playground.

Runners view

Display process health, PID, engine version, run state, scheduler phase, worker count, queue saturation, QPS/RPS, p50/p95/p99, errors, timeouts, open connections, CPU, memory, and last heartbeat. Controls include graceful stop and force kill. Force kill is enabled only after cancellation is requested or the heartbeat is stale.

Kill-switch contract

  1. Stop: send CancelRun(Graceful) and show draining progress.
  2. Terminate: after the grace deadline, terminate the child process.
  3. Kill: after a second deadline, use the platform process-kill API and mark the run interrupted.

Browser/WASM mode cannot run arbitrary traffic because browser networking APIs do not expose raw TCP, UDP, TLS, or packet control. It remains an editor and report surface unless connected explicitly to a local engine process.

Portable Distribution

Rust remains the selected implementation language. Portability is delivered through a release matrix: one self-contained binary per operating system and CPU target. A Linux binary cannot run on macOS or Windows, and a macOS binary cannot run on Linux.

Artifact Target Purpose
macOS desktop app Universal arm64 + x86_64 Tauri app plus packaged wiresurge sidecar.
macOS CLI arm64 + x86_64 universal Standalone CLI and internal engine mode.
Linux CLI x86_64-musl + arm64-musl Static-oriented binaries without a required OpenSSL installation.
Windows CLI x86_64 MSVC; arm64 later Standalone executable with Windows control-event shutdown handling.
  • CLI builds do not depend on Tauri, Node.js, or a browser runtime.
  • TLS uses a Rust implementation; certificate-root source is configurable as native, embedded, or file-based.
  • Release CI builds, tests, signs where applicable, and publishes checksums for every target artifact.
  • WASM is not a substitute for the native CLI because it cannot provide the required network and process capabilities.

Safety Model

The application can generate meaningful traffic. Safety controls should be first-class, especially because the CLI can run from automation.

Target safety

  • Workspace target allowlists.
  • Max QPS and max duration defaults.
  • Explicit confirmation for public IP targets above configured thresholds.
  • Dry-run mode that resolves variables and prints planned traffic.

Secret safety

  • OS keychain storage.
  • Redacted by default in UI, logs, and reports.
  • Secret access audit log inside local workspace metadata.
  • No secrets in workflow files unless intentionally marked unsafe.

Do not optimize away safety prompts in early versions. Make bypasses possible for CI and trusted runners, but require explicit configuration such as --yes, target allowlists, or signed run profiles.

Positioning

Position WireSurge as a controlled traffic workbench for owned systems. Public examples should emphasize repeatability, protocol inspection, safety limits, and explicit authorization rather than raw request volume alone.

Roadmap

  1. Runtime foundation: supervisor, lifecycle state machine, root cancellation token, bounded channels, and engine handle API.
  2. Async HTTP core: Rust TLS, HTTP/1.1 pool, HTTP/2 multiplexing, connection policy, redirects, and request timelines.
  3. Corpus service: memory-mapped inputs, offset index, deterministic random selection, unique permutation mode, bounded buffers.
  4. Desktop sidecar: local IPC, asynchronous events, Runners view, heartbeat monitoring, graceful stop, terminate, and force kill.
  5. Load engine: work admission, per-worker metrics, auto-ladder dimensions, stop conditions, and reproducible scheduler traces.
  6. DNS and protocol stages: UDP/TCP/DoT, EDNS0, PROXY protocol, TLS SNI, and shared assertions.
  7. Durability and security: atomic reports, crash recovery, keychain secrets, redaction, Git metadata, and safety limits.
  8. Extensibility: WASM plugins, advanced transports, packet templates, pcap replay, and optional high-rate native backends.

Open Questions

  • Must corpus randomization sample with replacement, visit every row exactly once, or support both modes?
  • What should the default connection experiment be: pooled, fresh per request, or explicit-only?
  • What are the default graceful-drain and force-kill deadlines?
  • Should Windows ship in the first release or after macOS and Linux behavior is stable?
  • Should reports be committed by default, or should the app ask per run?
  • Which HTTP imports matter first: curl, OpenAPI, Postman, Bruno, Insomnia, or Yaak?
  • How aggressive should the default safety limits be for public internet targets?
